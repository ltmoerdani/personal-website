# Internationalization (EN/ID) Architecture

**Topic:** architecture / i18n
**Created:** 2026-07-05
**Updated:** 2026-07-05

## Overview

The site is bilingual: **English (default)** and **Bahasa Indonesia**. The system uses **URL-based locale routing** powered by [`next-intl`](https://next-intl-docs.vercel.app/) v4. Every route is prefixed with a locale segment (`/en/...`, `/id/...`), and only the matching language is rendered server-side. This makes the language choice **shareable, SEO-crawlable, and persistent in the URL** — replacing the previous client-side CSS-toggle approach.

---

## How It Works

### 1. URL-Based Locale Routing

Each locale gets its own URL prefix:

| URL | Language | `<html lang>` |
|-----|----------|---------------|
| `ltmoerdani.com/en/...` | English | `en` |
| `ltmoerdani.com/id/...` | Bahasa Indonesia | `id-ID` |
| `ltmoerdani.com/` | → redirects based on `Accept-Language` | — |

Visiting the root `/` triggers locale negotiation via the middleware, which checks the `Accept-Language` header and the `NEXT_LOCALE` cookie, then redirects to the matching prefix (`/en` by default).

### 2. Folder Structure

```
app/
├── layout.tsx              # Pass-through root (Next requires it)
├── globals.css             # Design tokens (no more data-lang rules)
├── sitemap.ts              # Auto sitemap with hreflang alternates
├── robots.ts               # Robots → sitemap
└── [locale]/
    ├── layout.tsx          # <html lang>, metadata, fonts, NextIntlClientProvider
    ├── page.tsx            # Homepage
    ├── not-found.tsx       # 404 page
    └── case/
        ├── myarchery/page.tsx
        ├── opencode/page.tsx
        └── pro-archery/page.tsx

i18n/
├── config.ts               # Source of truth: locales, defaultLocale, siteBaseUrl
├── request.ts              # Server-side message loader (next-intl)
└── navigation.ts           # Locale-aware Link / useRouter / usePathname

messages/
├── en.json                 # Framework-level strings (lang name)
└── id.json

middleware.ts               # Locale negotiation + prefix enforcement
```

### 3. Middleware — Locale Negotiation

`middleware.ts` uses `createMiddleware` from `next-intl/middleware`:

- **Always** requires locale prefix (`/en/...`, `/id/...`)
- Visiting `/` negotiates via `Accept-Language` + `NEXT_LOCALE` cookie → redirects
- Persists choice in `NEXT_LOCALE` cookie for repeat visits
- Excludes Next internals and static assets (files with extensions)

### 4. The `<T>` Component — Hybrid Approach

The `<T en="..." id="..." />` API is **preserved** for ergonomic inline authoring. The difference is in how it renders:

**Before (CSS-toggle):** Both languages rendered in DOM, CSS hid one.
**Now (SSR single-render):** Only the matching language is sent to the browser, based on `useLocale()` from next-intl.

```tsx
// components/T.tsx
'use client';
import { useLocale } from 'next-intl';

export function T({ en, id }: { en: ReactNode; id: ReactNode }) {
  const locale = useLocale();
  return <>{locale === 'id' ? id : en}</>;
}
```

**Why hybrid?** The site has ~200 bilingual strings across 20+ components. Migrating all of them to JSON key-lookups (`t('hero.headline')`) would touch every component for marginal benefit on a solo portfolio. The `<T>` inline pattern keeps copy co-located with the component that uses it, while URL-based routing handles SEO, shareability, and persistence.

The `messages/*.json` files are intentionally minimal (framework-level strings only). If future content needs to migrate to message keys, the infrastructure is ready.

### 5. The `<LangToggle>` Button

Now navigates to the same path under the other locale prefix, instead of mutating `localStorage` and a `data-lang` attribute:

```tsx
// components/LangToggle.tsx
const router = useRouter();
const pathname = usePathname();
const nextLocale = locale === 'en' ? 'id' : 'en';

const toggle = () => {
  router.replace(pathname, { locale: nextLocale });
};
```

The choice lives in the URL — shareable, bookmarkable, and crawlable.

### 6. Navigation — Locale-Aware Links

Internal links use the locale-aware helpers from `i18n/navigation.ts`:

```tsx
import { Link } from '@/i18n/navigation';

// Automatically prepends the active locale:
<Link href="/case/myarchery">…</Link>
// On /en  → /en/case/myarchery
// On /id  → /id/case/myarchery

// Explicit locale switch:
<Link href="/" locale="id">Switch to ID</Link>
```

Never manually prepend `/en/` or `/id/` — pass the localeless path and let the helper resolve it.

## SEO Implementation

### `<html lang>` Attribute

Set dynamically per locale in `app/[locale]/layout.tsx`:
- `en` → `<html lang="en">`
- `id` → `<html lang="id-ID">`

### Hreflang Alternates

Every page declares all locale variants in `<head>`:

```html
<link rel="alternate" hrefLang="en" href="https://ltmoerdani.com/en" />
<link rel="alternate" hrefLang="id" href="https://ltmoerdani.com/id" />
<link rel="alternate" hrefLang="x-default" href="https://ltmoerdani.com/en" />
```

Generated via `generateMetadata()` in `app/[locale]/layout.tsx`.

### Sitemap (`app/sitemap.ts`)

Each route is emitted once per locale with hreflang alternates:

```xml
<url>
  <loc>https://ltmoerdani.com/en</loc>
  <xhtml:link rel="alternate" hreflang="en" href=".../en" />
  <xhtml:link rel="alternate" hreflang="id" href=".../id" />
  <xhtml:link rel="alternate" hreflang="x-default" href=".../en" />
</url>
```

Routes covered: `''` (home), `/case/myarchery`, `/case/opencode`, `/case/pro-archery`.

### Robots (`app/robots.ts`)

```txt
User-agent: *
Allow: /
Sitemap: https://ltmoerdani.com/sitemap.xml
```

### OpenGraph

`og:locale` is set dynamically:
- `en` → `en_US`
- `id` → `id_ID`

## Configuration

### `i18n/config.ts` — Source of Truth

```ts
export const locales = ['en', 'id'] as const;
export type Locale = (typeof locales)[number];
export const defaultLocale: Locale = 'en';
export const siteBaseUrl =
  process.env.NEXT_PUBLIC_SITE_URL ?? 'http://localhost:3000';
```

To add a new language: add it to `locales`, create `messages/{locale}.json`, add `localeHtmlLang` entry. Everything else adapts automatically.

### Environment Variable

Set in production (`.env` or hosting platform):

```bash
NEXT_PUBLIC_SITE_URL=https://ltmoerdani.com
```

Until set, the default `http://localhost:3000` is used (fine for dev).

## Persistence

| Storage | Key | Purpose |
|---------|-----|---------|
| Cookie | `NEXT_LOCALE` | Set by middleware after first visit; skips negotiation on repeat visits |
| URL segment | `/en/` `/id/` | Authoritative source of truth — shareable, bookmarkable |

No `localStorage` involved anymore. The previous `ltm-lang` key is deprecated.

---

## Adding a New Translatable String

Same as before — use the `<T>` component inline:

```tsx
import { T } from '@/components/T';

<T en="English text" id="Teks Indonesia" />

// With formatting:
<T
  en={<>Build the <strong>right</strong> product</>}
  id={<>Bangun produk yang <strong>tepat</strong></>}
/>
```

Never hardcode user-facing text without a `<T>` wrapper.

## What Changed from the Previous System

| Aspect | Before (CSS-toggle) | Now (URL routing) |
|--------|---------------------|-------------------|
| Language switching | `localStorage` + `data-lang` on `<html>` | URL segment (`/en`, `/id`) |
| DOM rendering | Both languages rendered, CSS hides one | Only matching language rendered (SSR) |
| SEO | Crawlers saw English only | Both languages fully crawlable |
| Shareability | `/case/myarchery` (no language in URL) | `/id/case/myarchery` (explicit) |
| Persistence | `localStorage` (`ltm-lang` key) | URL + `NEXT_LOCALE` cookie |
| `<T>` API | Unchanged | Unchanged (same `en` / `id` props) |
| Payload size | Larger (both languages in DOM) | Smaller (single language) |
| Library | None (custom) | `next-intl` v4 |

## Limitations

1. **`messages/*.json` are minimal** — the bulk of bilingual copy lives inline via `<T>`. If translator handoff becomes necessary, migrate copy to message keys.

2. **Warning: "middleware" deprecated in Next.js 16** — Next.js 16 renamed `middleware.ts` to `proxy.ts`. The functionality is identical; migration is cosmetic and non-urgent.

3. **Testimonial quotes remain English-only** — Original Fiverr reviews were in English. The `<T>` component's `id` variant falls back to English for untranslated quotes.
