## Overview

This document traces the conversion of the ltmoerdani.com portfolio from a self-contained bundled HTML prototype (`prototype/v3/`) to a production Next.js 16 App Router project. It covers what was extracted, how the bundler format was decoded, and the architectural decisions made during conversion.

---

## Source: The v3 Bundle Format

The prototype `prototype/v3/ltmoerdani standalone v2.html` is **not** a normal HTML file. It's a **self-extracting bundle** generated by a design tool's "dc-runtime" engine. Understanding its structure was the first challenge.

### Bundle Anatomy

```
ltmoerdani standalone v2.html
├── <script type="__bundler/manifest">    ← JSON: base64-encoded assets (gzip)
├── <script type="__bundler/template">    ← JSON: the actual HTML template
├── <script type="__bundler/ext_resources"> ← JSON: external resource mapping
└── <script>                              ← The unpacker runtime (dc-runtime)
```

| Asset UUID | Type | Compressed | What it is |
|------------|------|------------|------------|
| `ab3001a6-...` | `image/png` | No | Profile photo (703 KB) |
| `1a524b98-...` | `text/javascript` | Yes (gzip) | dc-runtime React app (61 KB) |
| `6bd2e3ee-...` | `font/woff2` | No | Archivo Vietnamese subset |
| `fdadbdeb-...` | `font/woff2` | No | Archivo Latin subset |
| `5d83cffa-...` | `font/woff2` | No | Instrument Serif Latin-ext |
| `795e6d4c-...` | `font/woff2` | No | JetBrains Mono Latin |
| *(+ 10 more font subsets)* | | | |

### How the Bundle Works at Runtime

1. On `DOMContentLoaded`, the unpacker reads the manifest
2. Base64-decodes each asset, gunzips if `compressed: true`
3. Creates a `blob:` URL for each asset
4. String-replaces all UUIDs in the template with their blob URLs
5. Strips `integrity` and `crossorigin` attributes (blob:null origin)
6. Parses the template via `DOMParser` and swaps `<html>`
7. Re-creates all `<script>` tags so they execute (DOMParser makes them inert)
8. Triggers `Babel.transformScriptTags()` for any `text/babel` blocks

### The x-dc Custom Element

Inside the template, the actual page content lives inside an `<x-dc>` custom element:

```html
<x-dc>
  <helmet>
    <!-- @font-face declarations for all font subsets -->
    <style>/* CSS variables, keyframes, language rules */</style>
  </helmet>

  <div data-screen-label="ltmoerdani.com">
    <!-- All visible page content with inline styles -->
  </div>

  <script type="text/x-dc" data-dc-script data-props='{...}'>
    class Component extends DCLogic {
      componentDidMount() { /* IntersectionObserver, counters */ }
      renderVals() { return { toggleLang: () => {...} } }
    }
  </script>
</x-dc>
```

The `data-props` attribute defines editor-configurable properties (accent color, photo shape, grain toggle) — these were hardcoded to defaults during conversion.

## Extraction Process

### Step 1: Decode Assets

```bash
# Profile photo (uncompressed base64 PNG)
node -e "
const entry = manifest['ab3001a6-...'];
fs.writeFileSync('public/profile.png', Buffer.from(entry.data, 'base64'));
"

# dc-runtime JS (gzip-compressed base64)
node -e "
const entry = manifest['1a524b98-...'];
const bin = Buffer.from(entry.data, 'base64');
fs.writeFileSync('/tmp/v3-app.js', zlib.gunzipSync(bin));
"
```

### Step 2: Extract Template Body

The template is a JSON string. Parsing it gives the full HTML document. The `<body>` contains the `<x-dc>` block with all page content.

### Step 3: Extract Logic

The dc-runtime JS is a React app that:
- Parses the `<x-dc>` template
- Renders it via `React.createElement`
- Runs `IntersectionObserver` for scroll reveals
- Runs counter animations for `[data-count]` elements
- Provides `toggleLang` for EN/ID switching

### Step 4: Case Study Files

Three additional `.dc.html` files in `prototype/v3/` contain case study pages:
- `Case Pro Archery.dc.html` — dark hero, 4-phase roadmap
- `Case MyArchery.dc.html` — green tint hero, domain authority
- `Case OpenCode.dc.html` — dark hero, terminal mockup

Each follows the same `<x-dc>` structure but with simpler CSS (no full font @face blocks — they link Google Fonts instead).

## Conversion Decisions

### 1. Inline Styles → React Style Objects

The prototype uses pervasive inline `style="..."` attributes. We preserved this approach in React via `style={{...}}` objects rather than converting to CSS modules or Tailwind.

**Rationale:**
- The prototype's visual fidelity is in the inline styles
- Converting to CSS classes would require naming 200+ utility classes
- React inline styles are type-safe with TypeScript
- No CSS framework dependency to maintain

**Trade-off:** Hover states (`style-hover` in the prototype) can't be done with inline styles in React. These were converted to `onMouseEnter`/`onMouseLeave` handlers, requiring `'use client'` on affected components.

### 2. The `<T>` i18n Component

The prototype uses `<span data-l="en">...</span><span data-l="id">...</span>` pairs, with CSS hiding the inactive language based on `<html data-lang>`.

We wrapped this pattern in a reusable component:

```tsx
// Before (prototype):
<span data-l="en">See the work</span><span data-l="id">Lihat karya</span>

// After (Next.js):
<T en="See the work" id="Lihat karya" />
```

The CSS visibility rules in `globals.css` are unchanged — they still key off `html[data-lang]`.

### 3. Scroll Reveal → `<Reveal>` Component

The prototype's `IntersectionObserver` logic was extracted into a `<Reveal>` wrapper:

```tsx
<Reveal index={0}>
  <h2>Section title</h2>
</Reveal>
```

The `index` prop controls stagger delay (multiplied by 110ms).

### 4. Language Toggle → `<LangToggle>` Component

The prototype's `toggleLang` function (inside dc-runtime) became a self-contained client component that:
- Reads `localStorage['ltm-lang']` on mount
- Toggles `document.documentElement.dataset.lang`
- Persists the choice to `localStorage`
- Shows the *opposite* language label (the one you'd switch to)

### 5. Counter Animation → `<Counter>` Component

The `[data-count]` counter logic became a `<Counter>` component:

```tsx
<Counter target={12000} suffix="+" fallback="12,000+" />
```

The `fallback` prop is critical for SSR — it shows the formatted string before hydration, preventing layout shift.

### 6. Photo Shape & Grain → CSS Attribute Selectors

The prototype's editor-configurable `photoShape` and `grain` props map to `html[data-photoshape]` and `html[data-grain]` attribute selectors in `globals.css`. These are set once and don't need runtime toggles in the production site (defaults: circle photo, grain on).

## Font Loading

### Prototype: Self-Hosted @font-face

The bundle includes 16 `@font-face` declarations with base64-embedded woff2 subsets (Vietnamese, Latin-ext, Latin, Cyrillic, Greek) for Archivo, Instrument Serif, and JetBrains Mono.

### Next.js: `next/font/google`

We replaced self-hosting with `next/font/google` in `app/layout.tsx`:

```tsx
const archivo = Archivo({
  subsets: ['latin'],
  weight: ['400', '500', '600', '700', '800', '900'],
  display: 'swap',
  variable: '--font-archivo',
});
```

**Benefits:**
- Automatic font optimization (subset, preload, fallback)
- No base64 bloat in the HTML
- CSS variables exposed: `--font-archivo`, `--font-instrument`, `--font-jetbrains`
- Zero layout shift via `display: 'swap'`

**Trade-off:** Cyrillic/Greek/Vietnamese subsets dropped (not needed for an English/Indonesian site).

## What Was NOT Converted

| Prototype Feature | Status | Reason |
|-------------------|--------|--------|
| Editor config (`data-props`) | Hardcoded | No design tool integration in production |
| Babel standalone | Removed | Next.js handles JSX compilation |
| dc-runtime React app | Replaced | Native Next.js App Router replaces it |
| `style-hover` attribute | Converted to `onMouseEnter/Leave` | Not a web standard |
| Blazz case study page | Card only, no detail page | Not in v3 prototype |
| JAKI case study page | Card only, no detail page | Not in v3 prototype |
| Book (Validasi Ide Bisnis) | Card only, external link | Not a case study |
